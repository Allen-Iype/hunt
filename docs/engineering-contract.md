# Hunt — Engineering Contract

Binding rules for anyone (human or AI) implementing Hunt. The SDD
([architecture/software-design.md](architecture/software-design.md)) is the
architectural source of truth; this document governs *how* work is executed.

## Role

Implement the project while preserving the architecture. Do not redesign
unless implementation exposes a genuine architectural issue — in that case:
stop, explain the issue and why the current design is insufficient, propose
alternatives, and wait for approval. Never silently modify the architecture.

## Engineering principles

Local First · Capability Driven · Source Agnostic · AI Agnostic ·
Deterministic where possible · Structured Data First · Plugin Friendly ·
Testability · Simplicity over Cleverness · YAGNI.

If an implementation decision conflicts with these, stop and ask.

## Development rules

- Never implement work outside the current milestone; never skip milestones;
  never begin the next milestone automatically.
- Never refactor unrelated code while implementing a milestone.
- Never introduce abstractions "for the future" or dependencies without a
  demonstrated need.
- Every line of code should have an obvious reason to exist.

## Git rules

**Never execute any git command that changes repository state or history**
(commit, push, merge, rebase, cherry-pick, reset, stash, tag, …).
Read-only git commands (status, diff, log) are fine.

At the end of a milestone, ONLY suggest a clean conventional commit message
(e.g. `feat(storage): implement canonical Job repository`). No Co-authored-by,
no "Generated by"/AI attribution of any kind.

## Milestone workflow (every milestone, exactly this sequence)

1. **Explain** the milestone: objectives, architecture affected, deliverables,
   acceptance criteria.
2. **Plan**: files to create/modify, dependencies (justified), testing
   approach, risks.
3. **Implement.**
4. **Test**: unit, integration, contract tests where appropriate; regression
   tests for every bug fix.
5. **Validate**: project builds, all tests pass, no architectural violations;
   explain limitations.
6. **Document** (mandatory — no milestone is complete without it).
7. **Stop.** Wait for approval. Never continue automatically.

## Documentation duties

Maintain throughout implementation:

- `docs/implementation/progress.md` — update after every milestone
  (statuses, chronological log, current focus, next steps, tech debt, risks,
  blockers).
- `docs/implementation/roadmap.md` — milestone table + update log.
- `docs/implementation/changelog.md` — Added/Changed/Fixed/Deferred/Breaking
  per milestone.
- `docs/implementation/decisions.md` — every deviation from or refinement of
  the SDD (date, decision, reason, alternatives, impact, affected section).
- `docs/architecture/adr/` — a new numbered ADR for every new architectural
  decision; never rewrite history.
- `docs/implementation/known-issues.md`, `docs/testing.md`,
  `docs/contributing.md`, `README.md` — keep current.

## Testing requirements

Every module has tests. Every adapter satisfies its port's contract tests.
Every bug fix ships a regression test. AI tests are deterministic via
record/replay (never a live provider in CI). The no-AI suite must always
stay green.

## Completion checklist (all must be true, then STOP)

✅ Feature implemented · ✅ Tests written · ✅ Tests passing ·
✅ Documentation updated · ✅ Progress updated · ✅ Roadmap updated ·
✅ Changelog updated · ✅ Decision log updated (if needed) ·
✅ ADR created (if needed) · ✅ Clean commit message suggested.
